# تحسينات إضافية

## 1) فهارس قاعدة البيانات + جدول الحظر
شغّل الملف ده مرة واحدة على قاعدة بياناتك (بعد ما تكون شغّلت `database.sql` الأصلي):
```
mysql -u USER -p DATABASE < migration_block_and_indexes.sql
```
ده هيضيف:
- فهرسين مركّبين لتسريع استعلامات "غير المقروء" (`idx_receiver_read`, `idx_conv_read`)
- جدول `blocked_users` الجديد

## 2) ميزة حظر المستخدمين
- زرار "⋮" في هيدر الشات فيه خيار حظر/إلغاء حظر.
- لو فيه حظر بين طرفين (في أي اتجاه)، `send_message.php` بيرفض الإرسال تلقائيًا.
- ملفات جديدة: `block_user.php`, `block_status.php`، ودوال في `User.php`
  (`blockUser`, `unblockUser`, `isBlocked`, `isBlockedEitherWay`, `getBlockedUsers`).
- ملحوظة: لسه محتاج (اختياري) تضيف صفحة "قائمة المحظورين" في الإعدادات لو حبيت
  المستخدم يشوف/يدير القائمة من مكان واحد بدل ما يفتح كل محادثة لوحدها.

## 3) نظام Logging (`config/logger.php`)
بدل `error_log()` المباشر، بقى فيه كلاس `Logger` بيكتب في
`storage/logs/YYYY-MM-DD.log` (محمي بـ `.htaccess` من الوصول من المتصفح).
استخدمه في أي مكان محتاج تسجيل حدث أو خطأ:
```php
require_once __DIR__ . '/../../../../config/logger.php';
Logger::info('User logged in', ['user_id' => $id]);
Logger::error('Something failed', ['detail' => $e->getMessage()]);
Logger::exception($e);
```

## لسه ناقص (مقترح للمرحلة الجاية)
- Real-time عبر WebSockets بدل الـ polling.

# جولة تانية من التحسينات

## 4) صفحة إدارة المستخدمين المحظورين
- صفحة جديدة: `app/Modules/Profile/Views/blocked_users.php`
- رابط ليها من صفحة البروفايل الشخصي بجانب "تعديل الملف".
- بتعرض كل المحظورين وتسمح بإلغاء الحظر من مكان واحد.

## 5) حذف الرسائل من الواجهة
- الدالة كانت موجودة بالفعل في الموديل بس مش مربوطة بأي مكان.
- endpoint جديد: `app/Modules/Chat/Api/delete_message.php`
- في الشات: بلاقي زرار "×" أحمر صغير بيظهر عند الـ hover على أي رسالة بعتّها إنت (مش على رسائل الطرف التاني).
- إصلاح جانبي: `deleteMessage()` في الموديل كانت بترجع "نجاح" حتى لو محذفتش حاجة فعليًا (لو حد حاول يمسح رسالة مش بتاعته)؛ دلوقتي بتتحقق من `affected_rows` فعليًا.
- إصلاح جانبي تاني: `MessageController.php` كان بيعمل `require` لمسار بحروف صغيرة (`models/`) بينما الفولدر الفعلي `Models/` بحرف كابيتال — ده كان هيفشل تمامًا على أي سيرفر Linux حساس لحالة الأحرف. اتصلح.

## ملحوظة عن كلمة المرور المنسية
لاحظت إن فورم "تغيير كلمة المرور" في `edit_profile.php` بيعمل POST مباشر لـ
`AuthController.php` وهو كلاس PHP عادي مش endpoint، فالفورم ده مش هيشتغل فعليًا
حالياً. ده يحتاج API endpoint مخصص لتغيير كلمة المرور (زي باقي الـ APIs) — قولي
لو عايزني أضيفه في الجولة الجاية.

# إصلاح مشكلة "الصفحة فاضية / ERR_EMPTY_RESPONSE"

كان فيه خطأ من الأساس (موجود في الكود الأصلي قبل أي تعديل مني) في `.htaccess`
الرئيسي: أسطر `php_flag` و`php_value` كانت مكتوبة برّه أي `<IfModule>`. لو
الاستضافة (زي InfinityFree غالبًا) بتشغّل PHP كـ FastCGI مش كموديول `mod_php`
في أباتشي، فأباتشي **يرفض ملف `.htaccess` بالكامل** ويرجّع صفحة فاضية تمامًا
(500 Internal Server Error بدون أي محتوى) لأي طلب تحت نفس المسار — وده بالظبط
شكل `ERR_EMPTY_RESPONSE`.

**الإصلاح:**
- لفيت أسطر `php_flag`/`php_value` جوه `<IfModule mod_php.c>` (ومتغيراتها 7/8) في `.htaccess`.
- ضفت ملف `.user.ini` جديد في جذر المشروع، وده البديل اللي بيشتغل فعليًا على استضافات FastCGI زي InfinityFree.

**بعد ما ترفع النسخة الجديدة:**
- استنى شوية (لحد 5 دقايق) لأن `.user.ini` بياخد وقت قبل ما يتفعّل (caching)، أو جرب تعمل "Reload" للموقع من لوحة تحكم الاستضافة لو متاح.
- لو المشكلة استمرت، افتح صفحة عادية زي `/index.php` مباشرة (مش فولدر) وشوف لو بتشتغل، عشان نتأكد المشكلة اتحلت من مصدرها.

# لوحة تحكم الأدمن + نظام الحظر الكامل

## الخطوات (اعملها بالترتيب)

1. **شغّل الـ migration** على قاعدة بياناتك:
   ```
   mysql -u USER -p DATABASE < migration_admin_and_bans.sql
   ```
   ده هيضيف: أعمدة الحظر لجدول `users`، جدول `admins`، وجدول `ban_logs` (سجل كل عمليات الحظر).

2. **غيّر `ADMIN_SETUP_KEY`** في ملف `.env` لقيمة سرية من عندك (أي نص عشوائي طويل).

3. **افتح صفحة الإعداد مرة واحدة بس**:
   ```
   https://yourdomain.com/app/Modules/Admin/Views/setup.php
   ```
   حط فيها المفتاح السري + اسم مستخدم وباسورد الأدمن بتاعك. الصفحة دي بتتقفل تلقائيًا بعد أول حساب أدمن يتعمل (مينفعش حد يعمل حساب أدمن تاني من نفسها تاني).

4. **بعد كده ادخل من**:
   ```
   https://yourdomain.com/app/Modules/Admin/Views/login.php
   ```
   ده رابط خاص بيك مش هيظهر في أي مكان في الموقع العادي (مش موجود في أي قائمة تنقل)، وممنوع من الفهرسة في `robots.txt` وبـ meta tag `noindex`.

## اللوحة بتوريك
- إجمالي المستخدمين، عدد المتصلين، عدد المحظورين، عدد التسجيلات النهاردة.
- جدول بكل المستخدمين (اسم، رقم هاتف، تاريخ التسجيل، الحالة) مع بحث.
- زرار "حظر" بيدّيك تكتب سبب (اختياري)، وزرار "إلغاء حظر".

## الحماية المطبّقة
- **جلسة أدمن منفصلة تمامًا** (`$_SESSION['admin_id']`) عن جلسة المستخدمين العاديين — حتى لو حصل أي مشكلة في نظام المستخدمين العاديين، ده مش هيأثر على صلاحيات الأدمن.
- **Brute-force protection** منفصلة لدخول الأدمن (5 محاولات / 5 دقايق قفل).
- **CSRF** على كل الفورمات والـ API الخاصة بالأدمن.
- **الحظر يتفعّل فورًا وفي كل حتة بالموقع**: عدّلت `isLoggedIn()` في `config/session.php` نفسها (المستخدمة في كل صفحة وكل API) عشان تتحقق من `is_banned` من الداتابيز، فلو حد اتحظر وهو داخل فعلاً، أول طلب بعدها (فتح صفحة، إرسال رسالة، تحديث الشات...) هيتقفل تلقائيًا من غير ما يحتاج يعمل تسجيل خروج ودخول.
- **منع الدخول من الأساس** لو الحساب محظور: `login()` بترفض الدخول برسالة توضح سبب الحظر (بعد التأكد من صحة الباسورد، مش قبلها، عشان محدش يعرف إن رقم معين محظور من غير ما يعرف الباسورد بتاعه).
- **سجل كامل** لكل عملية حظر/إلغاء حظر في جدول `ban_logs` (مين عمل الإجراء وإمتى وليه).
- فولدر الأدمن كامل ممنوع من الفهرسة (`robots.txt` + `X-Robots-Tag` header + meta noindex).

## ملحوظة مهمة للأمان
- بعد ما تعمل حساب الأدمن، **غيّر قيمة `ADMIN_SETUP_KEY`** في `.env` لأي حاجة تانية (مش لازم تفضل زي ما هي)، كطبقة حماية إضافية.
- خزّن رابط `login.php` بتاع الأدمن لنفسك بس، ومتحطهوش في أي مكان عام.
- لو حبيت حماية إضافية (اختياري): تقدر تقفل الوصول لفولدر `/app/Modules/Admin/` كله إلا من الـ IP بتاعك عن طريق إضافة كده في `.htaccess` بتاع الفولدر:
  ```
  Order deny,allow
  Deny from all
  Allow from YOUR.IP.ADDRESS.HERE
  ```
  (بس ده هيبقى مشكلة لو الـIP بتاعك بيتغيّر، فسيبته اختياري).
